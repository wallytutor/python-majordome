1.1 Getting Started

If you do not intend to build the package, you can find us on Majordome on PyPI and install it with pip:

pip install majordome

To register majordome as a Jupyter kernel, run:

majordome --install-kernel

You can customize the kernel identity if needed:

majordome --install-kernel `
    --kernel-name majordome-dev `
    --display-name "Majordome (dev)"

1.2 Manual builds

If you don’t fit in the first group, you can build the package from source.

The environment for developing Majordome requires at least a reasonably mondern version of Python and Rust; documentation depends on Quarto and uv is used for managing the environment. The exact versions may change over time, but as of the last update, the following (minimum) versions are recommended:

  • Python 3.12+
  • Rust toolchain
  • Quarto CLI
  • uv
Tip

Ubuntu users can bootstrap the environment with the following commands; make sure to check the latest versions of the tools and update the commands accordingly.

sudo apt-get install -y python3-dev

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

curl -LsSf https://astral.sh/uv/install.sh | sh
source "$HOME/.local/bin/env"

quarto_repo="https://github.com/quarto-dev/quarto-cli"
quarto_deb="quarto-1.9.37-linux-amd64.deb"

wget $quarto_repo/releases/download/v1.9.37/$quarto_deb
sudo apt-get install -y ./$quarto_deb

The following steps illustrate how to build the package, generate the documentation, and publish it to GitHub Pages. The commands are provided for both Windows PowerShell and Unix-like shells; make sure to use the appropriate ones for your system.

  • Build the package:
uv build --wheel
  • Inspect the wheel (especially if new data was included):
python -m zipfile -l dist/*.whl
  • Create and a ctivate a virtual environment:
uv venv --python python3.12 .venv

# Activate on Windows:
. .venv/Scripts/activate

# Activate on Unix:
. .venv/bin/activate
  • Install the package (remove [docs] to expand the argument, then add it back to install the dependencies before generating the documentation):
uv pip install ./dist/*.whl[docs]
  • Point Quarto to the virtual environment:
# Windows:
$env:QUARTO_PYTHON = ".venv/Scripts/python.exe"

# Unix:
export QUARTO_PYTHON=".venv/bin/python"
  • Generate the docs:
quarto render docs

1.3 Developers

The development focuses on portability and ease of setup, so by default any dependency relying on a C/C++ compiler is forbidden. This means that the development environment can be set up on any platform without needing to install additional compilers or tools.

Majordome code is documented using Numpydoc style docstrings, and the documentation is generated by module majordome.documents using Quarto markdown. This design choice reflects the academic aim of the package, with reporting and notebook support as first-class citizens.

In addition to what is described in the manual builds section, developers can also need to publish the docs, update the version, and publish the new version to GitHub and PyPI. This is fully automated by release.py to ensure consistency in actions, simple run one of the following:

python release.py --version 1.0.0
python release.py --version patch
python release.py --version minor
python release.py --version major

Nonetheless, the following commands are documented to keep track of what is being automated. Please notice that everything related to virtual environment handling is ommitted for simplicity, as super-users should be able to handle it on their own:

# Publish the docs to GitHub Pages (owners only):
quarto publish gh-pages --no-prompt --no-browser docs

# Install (the first time) cargo-edit for Cargo.toml:
cargo install cargo-edit

# Manually set the new version:
# (remove `--dry-run` to actually update the version)
uv version 1.0.0 --dry-run
cargo set-version 1.0.0 --dry-run

# Or use the bump to automatically increment:
uv version --bump patch --dry-run
uv version --bump minor --dry-run
uv version --bump major --dry-run
cargo set-version --bump patch --dry-run
cargo set-version --bump minor --dry-run
cargo set-version --bump major --dry-run

# Push the new version to GitHub (DO NOT COPY THIS):
gh release create v1.0.0 dist/*.whl --generate-notes
  • Publish the new version to PyPI (owners only) is done manually with uv:
uv publish
